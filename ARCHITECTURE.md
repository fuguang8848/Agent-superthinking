## 一、背景与目标

### 已有体系

超思考 v5：路由 → 专家分析 → 反馈 → 画像更新（单轮独立输出）

### v6 升级目标

引入 **Multi-Agent 圆桌辩论**，让专家围绕问题进行真实的来回交锋，而不是各自独立输出。

核心改变：
- 专家动态参与，辩论过程中按需加入或离开
- 方法论作为工具池，供专家在辩论中调用
- 动态轮次 + 主持人判断收敛
- 分歧无法收敛时，主持人向用户询问更多信息

## 二、已确认的设计决策

### 2.1 辩论形式

**结构化圆桌辩论**（模型C）。

流程：
1. 主持人（LLM）选定初始专家组合
2. 第一轮发言（并行，各方独立陈述初始观点）
3. 主持人整理"论点菜单"（有效的可辩论论点）
4. 后续轮次：各方针对上一轮的具体论点选择性反驳/深化
5. 主持人判断收敛，或分歧无法收敛时向用户询问更多信息
6. 用户LLM综合会议结论和各方最终观点，输出最终判断

### 2.2 对话模式

**模型C：结构化辩论 + 自由补充**

每轮发言规则：
- **必须**：每位专家必须针对至少一个其他专家的具体论点发言
- 格式：`"针对[专家X]关于[论点Y]的判断，我认为..."`
- 不是回应所有论点，而是**选择性回应**核心分歧点
- 允许一位专家同时针对多位专家的不同论点发言
- **同时允许**：自由补充发言（不依附于任何具体论点，针对当前问题提出独立洞察）

### 2.3 主持人双重角色

主持人（LLM）把控全局，对内对外双重职责：

| 角色 | 职责 | 说明 |
|------|------|------|
| **对内：辩论组织者** | 维护辩论秩序，整理论点菜单，判断收敛 | 主持辩论全过程的正常运行 |
| **对外：整合咨询者** | 可以外部咨询专家，私下获取观点用于完善自己的综合判断 | 不直接参与辩论，但为结论提供更丰富的参考 |

### 2.4 动态专家参与模式

**专家参与有两种模式，由主持人判断使用哪种：**

| 模式 | 说明 | 场景 |
|------|------|------|
| **正式参与** | 专家全程参与辩论，遵循结构化规则 | 需要该视角的持续交锋 |
| **外部咨询** | 主持人私下询问专家，观点用来武装主持人的综合判断，不直接参与辩论 | 需要某个观点但不想打乱当前辩论节奏；主持人想完善自己的判断 |

**外部咨询的价值：**
- 不需要为了一个观点引入新专家打乱辩论节奏
- 主持人合结论点时，有更丰富的参考
- 主持人判断质量更高

### 2.5 动态专家池

**专家列表是流式的，不是固定的。**

辩论过程中，主持人按需调整阵容：

| 情况 | 主持人动作 |
|------|-----------|
| 当前专家组合缺少某个视角 | 决定用外部咨询，还是正式邀请该专家加入 |
| 某专家严重偏离论题 | 向用户提议将该专家请离会议 |
| 新专家加入（正式参与） | 主持人给他：当前核心论点 + 正在讨论的问题 |
| 专家主动退出 | 主持人通知会议，该专家不再参与后续讨论 |

**新专家加入（正式参与）时的上下文同步：**
- 不是读完整辩论历史（太重）
- 主持人给新专家：当前正在辩论的核心问题 + 需要他判断的具体论点
- 新专家直接针对给他的论点发表看法，之后正常参与后续轮次

### 2.6 论点菜单标准（主持人规则）

#### 2.6.1 有效论点判断标准

主持人每轮结束时，从专家发言中提取论点，形成下一轮"论点菜单"。

一个论点要被纳入菜单，必须满足：

| 标准 | 要求 | 排除 |
|------|------|------|
| **具体性** | 针对具体论点，不是泛泛而谈 | "X总体是好的"、"这个问题很复杂" |
| **可反驳性** | 有明确的判断，可以被支持或反驳 | "这个问题需要更多思考" |
| **论证性** | 有理由或证据支撑 | 只有观点没有理由 |
| **分歧性** | 与其他专家观点存在真实差异 | 和其他人说的一模一样 |

#### 2.6.2 论点菜单格式

主持人每轮输出标准格式的论点菜单：

```
## 第N轮论点菜单

### 核心论点（可反驳）
1. [专家A] 认为 [X]，因为 [Y]
2. [专家B] 认为 [P]，因为 [Q]
3. [专家C] 认为 [M]，因为 [N]

### 已收敛的论点（可选记录）
- [专家A] 和 [专家B] 都认为 [R]

### 建议下轮重点关注
- [专家C] 和 [专家D] 在 [S] 上分歧最深，可能需要进一步辩论
```

#### 2.6.3 主持人整理论点的规则

```
规则1：从每位专家发言中提取"观点 + 理由"的完整单元
规则2：只保留满足四标准的论点，过滤掉无实质内容的表态
规则3：论点编号供下轮发言引用，同一论点可能被多方引用
规则4：每轮菜单独立，论点不跨轮继承（但论点编号可追溯）
```

### 2.7 方法论定位

方法论**不参与辩论**，而是作为**工具池**供专家和主持人调用。

**专家用法**：
- 在辩论中说"我用博弈论检验一下我刚才的论点"
- 从方法论视角重新分析自己的论点是否成立

**主持人用法**：
- 提示专家："要不要用[方法论]的角度看一下？"
- 当某论点有漏洞时，引导专家使用方法论检验

**方法论索引**：`experts/methods/INDEX.md`
- 每个方法论有简介（这是什么 + 何时用）
- 详细介绍在各自目录的 SKILL.md 中

**乱用防护**：主持人有权提示"这个方法论不适用于当前的论点"

### 2.8 轮次机制

**动态轮次**：算法 + 主持人结合

主持人维护的客观数据：
- 论点重叠率（逐轮相似论点出现频率）
- 新增论点密度（本轮新提出的独特论点数量）
- 置信度变化（"可能/也许/我认为"等词的变化）

收敛信号：
- 论点重叠率上升 + 新增论点持续减少 → 可能收敛
- 最终由主持人综合判断是否继续

**最大轮次**：动态配置，默认5轮，用户可随时修改

### 2.9 分歧无法收敛时

**分歧无法收敛是常态，不是异常。**

当主持人判断分歧无法收敛时：
1. 主持人暂停辩论
2. 主持人向用户询问更多具体信息
3. 期望：用户提供的更多信息能帮助专家找到底层相同观点
4. 辩论继续，基于新信息

### 2.10 结论合成

三层信息源：

```
会议结论（辩论产生的共识点 + 分歧点 + 未解决的核心争议）
    +
各专家最终观点（经过辩论后的修正观点）
    +
外部咨询观点（主持人私下整合的专家意见，用于完善主持人判断）
    ↓
用户LLM → 综合三者 + 自己的判断 → 最终输出
```

### 2.11 触发机制

| 触发方式 | 实现 |
|---------|------|
| 自动触发 | 框架初始化时加载技能，LLM判断问题复杂时自动调用 |
| 手动触发 | 用户说"思考一下"/"深度分析"等触发词 |
| 成本提醒 | 调用时提示"本次将消耗多轮LLM调用，是否继续" |

### 2.12 成本

成本不计。质量优先。

## 三、会议完整流程

### 3.1 初始化阶段

```
用户提出问题
    ↓
主持人分解问题 → 确定核心争议点
    ↓
主持人选择初始专家组合（基于问题类型 + 用户画像）
    ↓
会议开始通知 → 各专家注册
```

### 3.2 第一轮发言（并行）

各专家独立陈述初始观点，不针对任何人，格式：
`"关于[问题]，从[专家视角]来看，我认为[X]，理由是[Y]"`

### 3.3 主持人整理论点菜单

主持人提取本轮所有具体论点，按标准过滤，形成"论点菜单"（见2.6节格式）

### 3.4 后续轮次辩论循环

```
FOR 每轮（直到收敛 或 达到最大轮次）:

    各专家基于论点菜单，选择针对哪些论点发言
        ↓
    发言格式要求：
    "针对[专家X]的论点[X]，我认为..."
        ↓
    主持人监控方法论工具调用
        ↓
    主持人计算收敛信号
        ↓
    主持人判断（对内）：
        - 收敛 → 进入最终陈述
        - 未收敛 + 未达最大轮次 → 继续
        - 未收敛 + 分歧过深 → 向用户询问更多信息
        - 未收敛 + 达到最大轮次 → 进入最终陈述
    主持人判断（对外）：
        - 是否需要外部咨询某个专家？
        - 是否需要调整专家阵容（加入/请离）？
```

### 3.5 最终陈述阶段

各专家做最终表态（并行），格式：
`"综合本轮辩论，我的最终观点是[X]，因为[Y]"`

### 3.6 会议结论生成

主持人输出：
```
【共识点】哪些论点专家们最终达成一致
【分歧点】哪些核心争议未能解决
【未解决的根本矛盾】分歧背后的底层假设差异
【建议】用户可以从哪些方向继续思考
```

### 4.2 收敛算法细节

| 指标 | 计算方式 | 状态 |
|------|---------|------|
| 论点重叠率 | 本轮与上一轮相同关键词/论点的比例 | 待定 |
| 新增论点密度 | 本轮新提出的独特论点数量 | 待定 |
| 置信度变化 | 发言中"可能/也许/我认为"等词的变化 | 待定 |
| 综合收敛信号 | 加权公式 | 待定 |

### 4.3 主持人仲裁触发条件

| 情况 | 主持人动作 |
|------|-----------|
| 收敛信号持续减弱 | 继续观察下一轮 |
| 分歧无法收敛且用户提供的新信息无助于解决 | 提示用户当前核心分歧在哪 |
| 当前专家组合缺少某个视角 | 决定用外部咨询还是正式邀请 |
| 某专家严重偏离论题 | 向用户提议将该专家请离 |
| 某专家连续2轮无新论点 | 主持人提示该专家引入新角度 |

### 4.4 最大轮次配置

- 默认值：5轮
- 存放位置：config.yaml
- 用户可随时修改

### 4.5 新专家加入时的上下文同步

| 问题 | 状态 |
|------|------|
| 上下文格式 | 核心论点 + 当前正在讨论的问题（不是完整历史） |
| 新专家发表初始观点还是直接参与当前轮？ | 发表初始观点后参与下一轮 |
| 外部咨询专家的上下文是什么？ | 主持人根据需要给他相关论点 |

## 五、不做的事

以下 v6 原提案已明确不做：

| 组件 | 原因 |
|------|------|
| SupervisorAdapter（复杂度分级） | LLM路由已自动选专家 |
| LearningsIntegration（经验闭环） | 客观性优先，避免过往经验影响判断 |
| ContextBoard（顺序接力） | 不是真正的辩论，只有 Model C 才算真正的多专家交锋 |
