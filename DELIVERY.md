## 一、项目背景

### 已有体系（v5）

```
问题 → 路由 → 专家分析 → 反馈 → 画像更新（单轮独立输出）
```

### v6 升级目标

引入 **Multi-Agent 圆桌辩论**，让专家围绕问题进行真实的来回交锋，而不是各自独立输出。

核心改变：
- 专家动态参与，辩论过程中按需加入或离开
- 方法论作为工具池，供专家在辩论中调用
- 动态轮次 + 主持人判断收敛
- 分歧无法收敛时，主持人向用户询问更多信息

---

## 二、核心流程

```
用户提出问题
    ↓
主持人初始化 → 选择专家组合
    ↓
第一轮：并行发言（各专家独立陈述初始观点）
    ↓
主持人整理论点菜单
    ↓
多轮辩论循环（直到收敛或达到最大轮次）
    ↓
最终陈述 + 会议结论
    ↓
用户LLM综合 → 最终输出
```

---

## 三、关键角色

### 3.1 主持人（Moderator）

主持人是整个辩论的核心协调者。

**职责：**
- 组织辩论（确保交锋，而不是各说各的）
- 整理论点菜单（每轮提取有效论点）
- 判断收敛（监控辩论走向，决定何时结束）
- 调整阵容（按需加入/请离专家，使用外部咨询）
- 对外咨询（私下询问专家，完善综合判断）
- 用户交互（主动向用户提问，获取信息）
- 方法论指引（提示专家使用方法论验证论点）

**主持人指令文档：** `MODERATOR.md`

### 3.2 专家（Expert）

专家是辩论的参与者，每个专家有独立的视角和思维方式。

**发言规则（每轮两条都要满足）：**

1. **必须：针对格式发言**
   ```
   针对[专家X]的论点[X]，我认为[Y]，因为[Z]。
   ```
   - 每位专家必须针对至少一个其他专家的具体论点发言
   - 允许同时针对多位专家的不同论点

2. **同时允许：自由补充**
   - 在针对格式发言之后，可以自由补充独立洞察
   - 格式：`"另外，我想补充一个观点..."`
   - 自由补充不依附于任何具体论点，同样可以被后续针对

### 3.3 方法论池（Methodology Pool）

方法论是检验工具，供专家和主持人调用。

**使用方法：**
- 专家：`我用博弈论检验一下我的论点`
- 主持人：`要不要用博弈论的角度看一下？`

**方法论索引：** `experts/methods/INDEX.md`
- 18个方法论的简介（这是什么 + 何时用）
- 详细介绍在各自目录的 SKILL.md

## 四、辩论结构

### 4.1 第一轮（并行）

各专家独立陈述初始观点，不针对任何人：

```
关于[问题]，从[专家视角]来看，我认为[X]，理由是[Y]。
```

### 4.2 论点菜单

主持人每轮结束后，提取本轮有效论点，形成"论点菜单"：

```
## 第N轮论点菜单

### 核心论点（可反驳）
1. [专家A] 认为 [X]，因为 [Y]
2. [专家B] 认为 [P]，因为 [Q]

### 已收敛的论点（可选记录）
- [专家A] 和 [专家B] 都认为 [R]

### 建议下轮重点关注
- [专家C] 和 [专家D] 在 [S] 上分歧最深
```

**论点有效性四标准：**

| 标准 | 要求 | 排除 |
|------|------|------|
| 具体性 | 针对具体论点，不是泛泛而谈 | "X总体是好的" |
| 可反驳性 | 有明确的判断，可以被支持或反驳 | "需要更多思考" |
| 论证性 | 有理由或证据支撑 | 只有观点没有理由 |
| 分歧性 | 与其他专家观点存在真实差异 | 和其他人说的一模一样 |

### 4.3 后续轮次（辩论循环）

```
FOR 每轮（直到收敛 或 达到最大轮次）:

    主持人宣布本轮论点菜单
        ↓
    专家基于论点菜单，选择针对哪些论点发言
        ↓
    主持人监控方法论工具调用
        ↓
    主持人计算收敛信号
        ↓
    主持人判断：
        - 收敛 → 进入最终陈述
        - 未收敛 + 未达最大轮次 → 继续
        - 未收敛 + 分歧过深 → 向用户询问更多信息
        - 未收敛 + 达到最大轮次 → 进入最终陈述
```

### 4.4 最终陈述

达到收敛条件或最大轮次后，各专家做最终表态：

```
综合本轮辩论，我的最终观点是[X]，因为[Y]。
```

### 4.5 会议结论

主持人输出：

```
【共识点】哪些论点专家们最终达成一致
【分歧点】哪些核心争议未能解决
【未解决的根本矛盾】分歧背后的底层假设差异
【建议】用户可以从哪些方向继续思考
```

## 五、动态专家池

**专家列表是流式的，不是固定的。**

辩论过程中，主持人按需调整阵容：

| 情况 | 主持人动作 |
|------|-----------|
| 当前专家组合缺少某个视角 | 决定用外部咨询，还是正式邀请该专家加入 |
| 某专家严重偏离论题 | 向用户提议将该专家请离会议 |
| 新专家加入（正式参与） | 主持人给他：当前核心论点 + 正在讨论的问题 |
| 专家主动退出 | 主持人通知会议，该专家不再参与后续讨论 |

**新专家加入时的上下文同步：**
- 不是读完整辩论历史（太重）
- 主持人给新专家：当前正在辩论的核心问题 + 需要他判断的具体论点
- 新专家发表初始观点后参与下一轮

## 六、专家参与模式

**两种模式，由主持人判断使用哪种：**

| 模式 | 说明 | 场景 |
|------|------|------|
| **正式参与** | 专家全程参与辩论，遵循结构化规则 | 需要该视角的持续交锋 |
| **外部咨询** | 主持人私下询问专家，观点用来武装主持人的综合判断，不直接参与辩论 | 需要某个观点但不想打乱当前辩论节奏 |

---

## 七、结论合成

三层信息源：

```
会议结论（辩论产生的共识点 + 分歧点 + 未解决的核心争议）
    +
各专家最终观点（经过辩论后的修正观点）
    +
外部咨询观点（主持人私下整合的专家意见，用于完善主持人判断）
    ↓
用户LLM → 综合三者 + 自己的判断 → 最终输出
```

## 十、不做的事

以下 v6 原提案已明确不做：

| 组件 | 原因 |
|------|------|
| SupervisorAdapter（复杂度分级） | LLM路由已自动选专家 |
| LearningsIntegration（经验闭环） | 客观性优先，避免过往经验影响判断 |
| ContextBoard（顺序接力） | 不是真正的辩论，只有结构化模型才算真正的多专家交锋 |
